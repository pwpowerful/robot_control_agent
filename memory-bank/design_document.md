# 智能机械臂控制 Agent 系统产品设计文档

## 1. 文档信息

- 文档名称：`design_document.md`
- 文档目标：将现有 PRD 收敛为可落地的单工位 MVP 产品设计方案
- 适用版本：MVP V1
- 文档语言：中文

## 2. 产品概述

### 2.1 背景

现有 PRD 提出了一个面向工业机械臂的智能 Agent 平台设想，希望通过大语言模型、视觉模型、知识检索和经验记忆能力，将自然语言指令自动转化为可执行的机械臂控制动作。原始设想覆盖多节点部署、多型号机械臂适配、经验沉淀和闭环验证，技术方向明确，但对于首版产品而言范围过大，不利于快速验证关键链路。

因此，本设计文档将首版产品收敛为一个面向单工位场景的 MVP，优先验证以下核心能力：

1. 用户能够通过 Web 控制台下达自然语言任务。
2. 系统能够基于视觉识别完成目标定位。
3. 系统能够自动生成并校验受控动作计划和执行脚本。
4. 系统能够在通过规则校验和轨迹检查后自动执行。
5. 系统能够通过视觉复检确认结果，并将成功经验写入长期记忆。

### 2.2 产品目标

MVP 的目标不是做成通用机械臂平台，而是打通一条完整、可审计、可验证的自动执行闭环，证明“自然语言到安全执行”的产品路径成立。

MVP 目标如下：

- 支持单工位、单品牌、单型号机械臂的定点抓取放置任务。
- 通过厂商 Python SDK 直连机械臂执行层。
- 通过 Web 控制台提供任务创建、任务状态查看、计划与结果追踪能力。
- 建立最小可用的安全机制，包括规则校验、轨迹检查、禁区校验、异常停机和审计追踪。
- 在视觉复检成功后自动沉淀经验，形成最小学习闭环。

### 2.3 非目标

以下内容明确不纳入 MVP：

- 多品牌、多型号机械臂统一适配。
- 多工位协同调度。
- ROS 优先驱动集成。
- 完整数字孪生仿真。
- 人工审批后执行。
- 自动异常重试。
- 无序料箱分拣和复杂柔性产线调度。

## 3. 版本范围与业务场景

### 3.1 版本范围

MVP 的业务边界如下：

- 部署形态：单机部署。
- 交互形态：Web 控制台。
- 机器人范围：单品牌单型号机械臂。
- 底层控制：厂商 Python SDK 直连。
- 核心任务：定点抓取放置。
- 执行模式：校验通过后自动执行。
- 安全策略：检测异常立即停机并告警。

### 3.2 首个业务场景

MVP 首先面向定点抓取放置任务，典型流程如下：

1. 操作员输入自然语言指令，例如“将工位 A 上的零件抓取并放到托盘 B 的指定位置”。
2. 系统识别目标物体和目标放置位置。
3. 系统结合视觉坐标、知识库约束和历史示教样本，生成动作计划。
4. 系统完成脚本生成与规则校验。
5. 系统自动执行。
6. 执行后通过视觉确认目标物是否已到达目标位置。
7. 任务成功则记录经验；失败则立即停机并生成告警。

### 3.3 成功标准

MVP 的验收优先关注以下指标：

- 任务成功率达到项目设定的试运行标准。
- 运行期间安全零事故。
- 所有任务均可审计追踪。
- 系统可稳定完成“输入指令 -> 自动执行 -> 视觉验收 -> 记录经验”的闭环。

## 4. 用户角色与使用方式

### 4.1 角色定义

#### 操作员

操作员是系统的一线使用者，主要负责：

- 创建任务。
- 查看任务执行状态。
- 查看当前执行计划、结果和告警。
- 在任务失败后查看失败原因并通知人工处理。

#### 管理员

管理员负责系统配置与规则维护，主要包括：

- 配置机械臂连接参数和工作空间参数。
- 维护工作禁区与安全规则。
- 维护知识库内容，包括 SDK 文档、SOP 和示教样本。
- 查看全量审计日志和告警记录。

### 4.2 Web 控制台使用范围

MVP Web 控制台至少包含以下能力：

- 任务创建与任务列表查看。
- 单任务详情查看。
- 执行计划与生成脚本查看。
- 实时执行状态与告警查看。
- 视觉验证结果查看。
- 知识库与示教样本管理。
- 机械臂配置与安全规则管理。
- 审计日志检索。

## 5. 产品能力设计

### 5.1 任务创建与理解

用户通过 Web 控制台输入自然语言任务。系统接收任务后，需要从指令中提取以下关键信息：

- 目标物体。
- 目标位置。
- 工位上下文。
- 当前激活的机械臂配置。
- 可执行任务类型。

若指令缺少关键要素，系统应在任务进入执行前给出结构化失败原因，而不是进入模糊执行。

### 5.2 VisionTool

VisionTool 是系统感知物理世界的入口，负责在任务执行前后提供视觉闭环支持。

MVP 中 VisionTool 包含两类能力：

#### 抓取前定位

- 扫描工作区域。
- 识别目标物体。
- 输出目标物体的物理坐标。
- 完成像素坐标到机械臂基坐标的映射。
- 识别是否存在安全风险目标，如人手进入禁区。

#### 执行后确认

- 在抓取放置完成后再次检测目标物。
- 确认目标物是否已经处于预期位置。
- 输出结构化确认结果，用于决定任务成功或失败。

### 5.3 Planner

Planner 负责将自然语言任务转化为受控动作步骤。MVP 中 Planner 只面向定点抓取放置场景，输出的逻辑步骤必须可被约束在预定义动作模板内。

典型步骤包括：

1. 定位目标物。
2. 接近目标物。
3. 闭合夹爪。
4. 提升目标物。
5. 转移到目标位置。
6. 放置目标物。
7. 松开夹爪。
8. 退出安全位置。

Planner 同时需要在规划前判断：

- 当前视野中是否存在人手等危险物体。
- 当前目标是否可定位。
- 当前目标位置是否落在允许工作区域内。

若任一条件不满足，则任务直接失败并触发告警。

### 5.4 Coder

Coder 负责将 Planner 输出的受控步骤转化为符合厂商 Python SDK 约束的执行脚本。

MVP 中 Coder 的设计原则如下：

- 不允许自由生成任意 Python 脚本。
- 必须限制在受控动作模板和厂商 SDK 白名单接口内。
- 输出结果需具备可审计性，能够标记脚本版本、生成时间、关联任务 ID。
- 每个动作步骤都应保留对应的输入参数和来源信息。

### 5.5 Critic

Critic 负责在执行前对计划和脚本进行校验。MVP 仅要求规则校验与轨迹检查，不做完整数字孪生。

校验内容包括：

- 脚本语法合法性。
- SDK 调用是否来自白名单。
- 关节角度是否超限。
- 位姿是否可达。
- 是否进入工作禁区。
- 基础防碰撞检查。
- 夹爪动作是否符合当前任务阶段约束。

Critic 的校验结果必须结构化输出，至少包括：

- 是否允许运行。
- 失败类别。
- 失败描述。
- 关联步骤。
- 建议回退模块。

若校验失败，系统不执行脚本，并记录为失败任务。

### 5.6 RobotTool

RobotTool 是对机械臂底层执行能力的封装层，MVP 通过厂商 Python SDK 直连。

RobotTool 需要负责：

- 接收通过校验的脚本或动作计划。
- 以受控方式调用厂商 SDK。
- 输出结构化执行状态。
- 捕获底层异常并映射为统一错误码。
- 在检测到异常时立即停机。

RobotTool 需要返回的状态至少包括：

- `Success`
- `Validation_Failed`
- `Collision_Detected`
- `Workspace_Violation`
- `Object_Not_Found`
- `Grasp_Slip`
- `Execution_Interrupted`
- `Emergency_Stop`

### 5.7 RAG 与记忆

#### 知识库范围

MVP 的知识来源限定为：

- 厂商 Python SDK 文档。
- 作业 SOP。
- 人工示教样本。

知识库的目的不是开放问答，而是为任务规划与脚本生成提供约束和参考。

#### 短期记忆

短期记忆用于保存当前任务上下文，例如：

- 当前任务输入。
- 当前工位环境状态。
- 当前识别到的目标与位置。
- 当前执行状态。
- 当前失败原因。

#### 长期记忆

长期记忆用于保存经过验证的成功经验，仅在视觉复检成功后写入。每条经验至少包含：

- 经验 ID。
- 关联任务 ID。
- 任务类型。
- 目标物类型。
- 关键抓取参数。
- 放置参数。
- 关联脚本版本。
- 视觉确认结果。
- 记录时间。
- 来源标识。

失败任务不得写入长期记忆。

## 6. 业务流程设计

### 6.1 端到端主流程

1. 操作员在 Web 控制台创建任务。
2. 系统创建任务记录并分配 `task_id`。
3. 系统加载当前激活机械臂配置和安全规则。
4. 系统从知识库中检索相关 SOP、SDK 约束和示教样本。
5. VisionTool 执行目标定位并返回视觉结果。
6. Planner 生成受控动作步骤。
7. Coder 生成受控 SDK 脚本。
8. Critic 执行规则校验与轨迹检查。
9. 若校验通过，RobotTool 自动执行。
10. 执行完成后，VisionTool 再次验证结果。
11. 若视觉确认成功，则任务完成并写入长期记忆。
12. 若执行失败或视觉确认失败，则任务失败、立即停机并生成告警。

### 6.2 失败处理流程

以下情况统一进入失败流程：

- 未识别到目标物。
- 识别到人手或危险物体。
- 脚本校验失败。
- 轨迹不可达。
- 碰撞风险。
- 执行中断。
- 夹爪滑落。
- 视觉复检失败。

失败处理规则如下：

1. 立即停止当前任务。
2. 机械臂进入安全停止状态。
3. 生成标准化告警事件。
4. 记录完整审计链路。
5. 不触发自动重试。
6. 不写入长期记忆。

## 9. 安全设计

### 9.1 核心安全原则

- 任何高风险异常优先于任务成功率。
- 任何未通过校验的脚本不得进入执行层。
- 任何危险目标检测结果都必须触发立即停机。
- 任何失败任务都不得写入长期记忆。

### 9.2 必须拦截的异常

- 人手进入工作区。
- 工作禁区入侵。
- 轨迹不可达。
- 关节超限。
- 防碰撞检查失败。
- 夹爪状态异常。
- 底层 SDK 调用异常。

### 9.3 停机策略

当检测到上述异常时，系统执行以下动作：

1. 立即终止当前执行。
2. 调用机械臂安全停止接口。
3. 写入停机事件。
4. 生成告警并通知控制台。
5. 将任务标记为失败。

## 10. 审计与可追溯设计

MVP 必须支持完整审计追踪。每一次任务至少记录以下链路：

- 谁在什么时间提交了什么指令。
- 系统解析出了什么任务目标。
- 系统加载了哪些知识条目和示教样本。
- Planner 产出了什么步骤。
- Coder 产出了哪个脚本版本。
- Critic 做了哪些校验以及结果如何。
- RobotTool 返回了什么执行状态。
- VisionTool 给出了什么复检结果。
- 是否写入长期记忆。

审计日志应支持按任务 ID、时间、机器人 ID、任务状态进行检索。

## 11. 部署与技术约束

### 11.1 MVP 部署方案

MVP 采用单机部署，核心原因如下：

- 降低系统集成复杂度。
- 更快打通首版执行闭环。
- 降低双节点通信、同步和排障成本。

系统逻辑上仍保持模块化拆分，便于后续演进为服务化架构。

### 11.2 模块边界

单机部署下建议保留以下逻辑模块：

- Web Console
- Task Service
- Vision Module
- Planner Module
- Coder Module
- Critic Module
- RobotTool Adapter
- Knowledge & Memory Service
- Audit & Alert Service

### 11.3 后续演进方向

后续版本可考虑从当前单机架构演进为以下形式：

- `Vision Service` 独立部署到感知算力节点。
- `Agent Service` 独立部署到认知与控制节点。
- 节点间通过 gRPC 等方式通信。

该部分仅作为未来演进路线，不属于 MVP 实现前提。

## 12. 验收与测试方案

### 12.1 功能验收场景

#### 正常流程

- 输入标准抓取放置指令。
- 系统成功完成定位、规划、校验、执行和视觉验收。
- 任务状态为成功，长期记忆写入成功。

#### 安全流程

- 视觉检测到人手或禁区入侵。
- 系统立即停机并产生告警。
- 任务状态为失败，不触发自动重试。

#### 校验拦截

- 生成计划或脚本存在关节超限、不可达或碰撞风险。
- 系统禁止执行，并记录失败原因。

#### 执行失败

- 抓取过程中出现滑落、执行中断或机械臂返回失败状态。
- 系统立即停机并记录失败，不写入长期记忆。

#### 记忆闭环

- 仅视觉验收成功任务会被写入长期记忆。
- 失败任务不会污染经验库。

#### 审计追溯

- 任一任务都能回放完整链路，包括指令、计划、脚本、校验、执行、视觉确认和结果。

### 12.2 核心验收指标

- 任务成功率。
- 安全零事故。
- 任务链路审计完整率。
- 视觉确认覆盖率。
- 失败任务拦截准确率。

## 13. 风险与缓解策略

### 13.1 主要风险

- LLM 输出不稳定导致计划质量波动。
- 视觉识别误检或漏检导致错误抓取。
- 厂商 SDK 封装不严导致执行层风险外溢。
- 自动执行模式下安全责任边界不清晰。
- 长期记忆写入质量不稳定，影响后续任务参考价值。

### 13.2 缓解策略

- 通过动作模板和 SDK 白名单约束 Coder。
- 通过 Critic 统一做规则校验与轨迹检查。
- 通过立即停机策略兜底高风险异常。
- 通过视觉复检决定是否写入长期记忆。
- 通过完整审计日志支撑问题追查和规则迭代。

## 14. 结论

本设计文档将原始 PRD 中偏平台化、偏未来态的能力收敛为一个单工位 MVP。该版本优先验证智能机械臂 Agent 的最小闭环：自然语言理解、视觉定位、规则约束下的计划和脚本生成、自动执行、视觉验收以及经验沉淀。

若 MVP 在任务成功率、安全性和审计可追溯性上达到预期，即可进入下一阶段，逐步演进为双节点部署、多型号适配和更复杂业务场景支持的正式产品。
