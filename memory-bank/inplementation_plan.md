# 智能机械臂控制 Agent 系统 MVP 实施计划

## 1. 使用规则

本计划面向 AI 开发者执行，必须严格遵守以下规则：

- 只以 `design_document.md` 和 `tech_stack.md` 作为实现基线。
- 每一步必须先完成本步骤的验证测试，再进入下一步。
- 每一步只解决一个局部问题，不允许跨步混做。
- 不允许引入 `tech_stack.md` 已明确不推荐的技术。
- 不允许在未完成结构化校验和安全约束前接入真实机械臂执行。
- 不允许跳过审计、告警、停机、验证测试。
- 本计划中的所有实现都应优先选择简单、可审计、可回放、可定位问题的方案。

## 2. 总体实施顺序

实施顺序分为六个阶段：

1. 基础工程阶段
2. 后端基础能力阶段
3. Agent 执行闭环阶段
4. Web 控制台阶段
5. 联调与测试阶段
6. 部署与验收阶段

只有当前阶段全部通过验证，才允许进入下一阶段。

## 3. 分步指令

### 步骤 01：冻结 MVP 范围与验收基线

#### 执行指令

- 通读 `design_document.md` 和 `tech_stack.md`，整理出一份实现边界清单。
- 明确本次只实现单工位、单品牌单型号、单机部署、Web 控制台、厂商 Python SDK 直连、自动执行、立即停机告警。
- 将所有非目标能力单独列出，作为后续实现排除清单。
- 整理一份“需求点 -> 模块 -> 验证方式”的追踪表，供后续每一步引用。

#### 交付物

- 一份范围清单
- 一份非目标清单
- 一份需求追踪表

#### 验证测试

- 检查追踪表是否覆盖 `design_document.md` 中的产品目标、核心模块、对象模型、安全设计、审计设计、测试场景。
- 检查非目标清单中是否明确包含多品牌、多工位、ROS 优先架构、完整数字孪生、自动重试。
- 检查是否没有引入 `tech_stack.md` 中明确不推荐的技术。

---

### 步骤 02：建立仓库目录结构

#### 执行指令

- 按单仓库方式创建顶层目录结构。
- 至少划分后端、前端、基础设施、测试、文档五类目录。
- 为每个目录补充职责说明，避免职责重叠。
- 不在本步骤加入业务实现，只搭建结构和命名规则。

#### 交付物

- 清晰的目录结构
- 每个目录的职责说明

#### 验证测试

- 检查目录结构是否能自然承载 `api-server`、`executor-worker`、`vision`、`robot_adapter`、`knowledge_memory`、`frontend`、`tests`。
- 检查是否不存在临时目录、模糊命名目录或重复职责目录。
- 检查任意新成员是否能仅通过目录名判断模块职责。

---

### 步骤 03：初始化后端工程基础

#### 执行指令

- 使用 `tech_stack.md` 规定的 Python 3.11 和 `uv` 初始化后端工程。
- 固定依赖管理方式、锁定文件策略和环境变量管理方式。
- 建立后端启动入口、配置入口和日志入口的基本约定。
- 明确开发环境、测试环境、生产环境的配置隔离方式。

#### 交付物

- 后端依赖清单
- 后端环境配置约定
- 后端启动与日志约定说明

#### 验证测试

- 在全新环境中验证后端依赖能够被一致安装。
- 验证后端应用可以启动到最小可运行状态。
- 验证配置缺失时，系统会给出清晰错误信息，而不是静默失败。

---

### 步骤 04：初始化前端工程基础

#### 执行指令

- 使用 `tech_stack.md` 规定的 Node.js、React、TypeScript、Vite、Ant Design 初始化前端工程。
- 建立页面目录、通用组件目录、API 调用目录、状态管理目录的边界。
- 统一路由命名规则、页面命名规则和状态管理命名规则。
- 不在本步骤实现具体业务页面。

#### 交付物

- 前端基础工程
- 前端目录与命名约定说明

#### 验证测试

- 验证前端项目可以本地启动。
- 验证基础路由框架可用。
- 验证页面结构支持后续接入任务、告警、审计、配置、知识库页面。

---

### 步骤 05：定义共享领域模型和任务状态机

#### 执行指令

- 根据 `design_document.md` 明确任务对象、执行计划对象、执行结果对象的字段定义。
- 定义统一的任务状态机，包括创建、规划、校验、待执行、执行中、验证中、成功、失败、急停。
- 为每个状态定义允许进入条件、允许退出条件和失败转移条件。
- 明确所有模块之间传递的数据结构名称和字段语义。

#### 交付物

- 领域模型说明文档
- 状态机说明文档

#### 验证测试

- 检查状态机是否覆盖 `design_document.md` 中的主流程和失败流程。
- 检查字段定义是否覆盖任务、计划、执行结果、告警、审计、长期记忆最小字段集。
- 检查状态转换是否不存在模糊状态或双重含义状态。

---

### 步骤 06：设计数据库结构与迁移策略

#### 执行指令

- 使用 PostgreSQL 16 设计核心表结构。
- 至少包含任务、执行计划、执行结果、告警、审计日志、知识条目、示教样本、长期记忆、系统配置、用户与角色表。
- 明确主键、外键、唯一约束、索引策略、审计字段和软删除策略。
- 定义数据库迁移策略与版本控制策略。

#### 交付物

- 数据模型设计说明
- 数据库迁移方案

#### 验证测试

- 在空数据库上验证迁移可以完整创建全部核心表。
- 验证核心外键关系正确，无法插入明显无效的引用关系。
- 验证关键查询字段具备索引支持，尤其是任务状态、任务时间、机器人标识、告警状态、审计检索字段。

---

### 步骤 07：建立审计与告警数据模型

#### 执行指令

- 单独设计审计事件模型与告警事件模型。
- 审计链路必须覆盖“指令 -> 计划 -> 校验 -> 执行 -> 视觉复检 -> 记忆写入”。
- 告警模型必须包含事件类型、严重级别、触发模块、关联任务、发生时间、处理状态。
- 明确哪些事件必须写入审计，哪些事件必须触发告警。

#### 交付物

- 审计事件模型说明
- 告警事件模型说明
- 审计与告警写入规则

#### 验证测试

- 检查审计事件是否能完整串起一次任务全链路。
- 检查危险检测、校验失败、执行异常、视觉复检失败是否都会生成告警。
- 检查普通状态变化不会被错误升级为高优先级告警。

---

### 步骤 08：实现配置管理与安全默认值

#### 执行指令

- 建立统一配置模块，管理数据库、LLM、视觉、机器人、鉴权、安全规则、日志等配置。
- 为所有安全相关配置设置保守默认值。
- 定义工作空间禁区、最大关节范围、紧急停机开关、模型超时、执行超时等默认配置项。
- 配置缺失时禁止进入真实执行流程。

#### 交付物

- 配置项清单
- 默认值清单
- 配置校验规则说明

#### 验证测试

- 验证缺少机器人配置时系统不能执行任务。
- 验证缺少安全规则时系统不能进入执行态。
- 验证明显非法配置会被系统拦截，而不是带着错误配置启动。

---

### 步骤 09：实现鉴权和角色权限模型

#### 执行指令

- 根据 `design_document.md` 实现操作员和管理员两类角色。
- 明确两类角色的页面访问权限、接口权限和数据操作权限。
- 使用服务端会话和 HTTP-only Cookie 方式设计认证。
- 管理员专属操作必须与操作员权限显式隔离。

#### 交付物

- 权限矩阵
- 鉴权流程说明

#### 验证测试

- 验证未登录用户无法访问业务接口和业务页面。
- 验证操作员不能修改机器人配置和安全规则。
- 验证管理员可以访问审计日志、配置和知识库管理能力。

---

### 步骤 10：实现后端基础 API 框架

#### 执行指令

- 建立统一 API 分组，包括任务、计划、脚本摘要、告警、审计、知识库、配置、认证。
- 统一请求响应格式、错误码格式和分页格式。
- 建立健康检查接口、版本接口和基础错误处理中间机制。
- 所有接口返回都必须结构化。

#### 交付物

- API 分组清单
- 统一响应规范
- 统一错误码规范

#### 验证测试

- 验证健康检查和版本接口稳定可用。
- 验证业务接口在异常情况下返回结构化错误，而不是返回不规则文本。
- 验证未实现接口不会暴露模糊占位响应。

---

### 步骤 11：实现任务创建与查询接口

#### 执行指令

- 实现任务创建、任务列表、任务详情、任务执行链路查询接口。
- 任务创建时必须校验输入完整性和最小可执行条件。
- 任务详情必须能展示原始指令、任务状态、目标物、目标位置、失败原因、创建人和时间。
- 所有任务状态变化都必须保留历史痕迹。

#### 交付物

- 任务接口
- 任务状态追踪能力

#### 验证测试

- 验证合法任务可以成功创建并生成唯一任务标识。
- 验证缺少关键字段的任务不能创建。
- 验证任务列表与任务详情中状态、时间和创建信息一致。

---

### 步骤 12：实现知识库、示教样本与长期记忆管理接口

#### 执行指令

- 实现知识条目管理、示教样本管理和长期记忆查询接口。
- 明确三类数据的字段差异和用途差异。
- 长期记忆写入逻辑只做接口和数据准备，不在本步骤连接执行流程。
- 知识条目必须支持检索元数据管理。

#### 交付物

- 知识条目接口
- 示教样本接口
- 长期记忆查询接口

#### 验证测试

- 验证三类数据可以独立创建、查询、更新。
- 验证长期记忆记录与知识条目不会混淆。
- 验证检索元数据字段完整可用。

---

### 步骤 13：实现执行队列与 Worker 骨架

#### 执行指令

- 按 `tech_stack.md` 建立“单后端进程 + 单执行 Worker”的执行模型。
- 使用数据库任务表和状态机驱动任务流转。
- 明确 Worker 只处理允许进入执行链路的任务状态。
- 明确 Worker 的重入保护、并发保护和任务锁定机制。

#### 交付物

- Worker 运行骨架
- 任务锁定与状态流转规则

#### 验证测试

- 验证同一任务不会被重复领取。
- 验证非法状态任务不会被 Worker 执行。
- 验证 Worker 异常退出后不会造成任务状态永久卡死。

---

### 步骤 14：实现 Vision 模块接口与模拟适配器

#### 执行指令

- 定义 Vision 模块接口，至少包含抓取前定位、危险目标检测、执行后视觉复检三类能力。
- 先实现模拟适配器，返回结构化结果，不接真实相机。
- 明确定义视觉输出字段，包括目标物识别结果、物理坐标、危险目标标识、复检结果。
- 所有结果必须携带置信度和时间戳。

#### 交付物

- Vision 模块接口定义
- Vision 模拟适配器

#### 验证测试

- 验证模拟适配器可以返回定位成功、危险目标检测成功、复检成功、目标未识别四类结果。
- 验证视觉输出字段完整可被 Planner 和执行流程消费。
- 验证危险目标结果会阻止流程继续前进。

---

### 步骤 15：实现 Robot 适配层与急停接口

#### 执行指令

- 定义厂商 Python SDK 的适配层边界。
- 明确允许调用的动作模板和白名单动作。
- 单独定义紧急停机接口和异常映射规则。
- 先实现模拟机器人适配器，不接真实机械臂。

#### 交付物

- Robot 适配层接口定义
- 白名单动作清单
- 急停规则说明

#### 验证测试

- 验证适配层只接受白名单动作。
- 验证非白名单动作会被拒绝。
- 验证模拟执行中可以触发成功、碰撞、滑落、中断、急停等结构化结果。

---

### 步骤 16：实现 Planner 服务

#### 执行指令

- 实现 Planner 的结构化输出能力，只允许输出定点抓取放置任务的受控步骤。
- 输出步骤必须限定在设计文档已定义的动作阶段内。
- 输入必须综合任务信息、视觉结果、知识检索结果和安全规则。
- Planner 不允许直接产出可执行脚本。

#### 交付物

- Planner 输入输出规范
- Planner 结构化结果

#### 验证测试

- 验证标准任务能够生成有限、清晰、可审计的步骤序列。
- 验证识别到人手、目标不可定位、目标超出工作区时，Planner 会返回失败而不是继续规划。
- 验证 Planner 不会输出设计范围之外的任务步骤。

---

### 步骤 17：实现 Coder 服务

#### 执行指令

- 实现 Coder，将 Planner 的结构化步骤映射为受控动作模板。
- Coder 只能从模板和白名单动作中组装执行内容。
- 输出必须保留任务标识、步骤映射关系、脚本版本和生成来源。
- 不允许自由生成任意 Python 行为。

#### 交付物

- Coder 映射规则
- 受控执行内容说明

#### 验证测试

- 验证相同步骤输入会得到一致的模板化输出。
- 验证超出白名单的动作请求无法通过 Coder。
- 验证生成结果能够追溯到任务、步骤和模板版本。

---

### 步骤 18：实现 Critic 校验链路

#### 执行指令

- 实现 Critic，对 Planner 和 Coder 的产物进行规则校验与轨迹检查。
- 至少覆盖语法合法性、动作白名单、工作禁区、关节限制、可达性、基础防碰撞和夹爪动作阶段约束。
- Critic 必须输出结构化校验结论、失败类别、失败描述和建议回退模块。
- 未通过 Critic 的任务严禁进入 Robot 适配层。

#### 交付物

- Critic 校验规则
- Critic 结构化结果格式

#### 验证测试

- 验证合法计划与模板能通过校验。
- 验证关节超限、不可达、禁区入侵、碰撞风险、非法夹爪阶段都会被拦截。
- 验证失败结果可明确定位到具体步骤或规则。

---

### 步骤 19：串联无硬件端到端执行闭环

#### 执行指令

- 使用任务接口、Vision 模拟适配器、Planner、Coder、Critic、Robot 模拟适配器和 Worker 串起完整流程。
- 在本步骤中禁止接真实相机和真实机械臂。
- 确保状态流转、审计记录和告警生成都贯通。
- 补充失败流转和回滚处理。

#### 交付物

- 无硬件端到端闭环
- 端到端状态流转记录

#### 验证测试

- 验证成功任务能够从创建流转到成功并产出完整审计记录。
- 验证危险目标、校验失败、执行失败、复检失败都能流转到失败或急停状态。
- 验证同一次任务的审计链路可从头追溯到尾。

---

### 步骤 20：实现执行后视觉复检与长期记忆写入规则

#### 执行指令

- 将执行后视觉复检接入完整闭环。
- 明确长期记忆写入前提为“视觉复检成功”。
- 写入长期记忆时保存任务类型、目标物、关键参数、脚本版本、复检结果、来源标识和时间。
- 失败任务禁止写入长期记忆。

#### 交付物

- 复检写入规则
- 长期记忆写入链路

#### 验证测试

- 验证复检成功任务会写入长期记忆。
- 验证复检失败任务不会写入长期记忆。
- 验证写入记录可追溯到原始任务和执行结果。

---

### 步骤 21：实现告警中心后端能力

#### 执行指令

- 实现告警列表、告警详情、停机事件查询能力。
- 区分普通失败与高优先级安全告警。
- 明确告警未处理、处理中、已处理状态。
- 将告警与任务、审计链路、触发模块关联起来。

#### 交付物

- 告警后端接口
- 告警状态模型

#### 验证测试

- 验证紧急停机事件会生成高优先级告警。
- 验证普通校验失败不会被误标为最高优先级。
- 验证告警详情可追溯到触发任务和相关审计记录。

---

### 步骤 22：实现 Web 控制台基础框架

#### 执行指令

- 建立控制台整体布局，包括导航、权限控制、页面壳、会话态处理和错误提示机制。
- 接入登录流程和角色区分逻辑。
- 明确全局加载、空状态、错误状态和权限拒绝状态的展示方式。
- 本步骤不实现完整业务页面细节。

#### 交付物

- 控制台基础框架
- 登录与权限壳

#### 验证测试

- 验证未登录状态下无法访问业务页面。
- 验证管理员和操作员看到的导航项不同。
- 验证全局错误与权限错误都有清晰界面反馈。

### 步骤 23：实现任务与执行链路页面

#### 执行指令

- 实现任务列表页、新建任务页、任务详情页。
- 任务详情页必须展示原始指令、目标物、目标位置、状态、计划步骤、脚本摘要、校验结果、执行结果、视觉复检结果、记忆写入状态和失败原因。
- 页面字段命名必须与后端对象模型一致。
- 重要状态变化必须可视化。

#### 交付物

- 任务列表页
- 新建任务页
- 任务详情页

#### 验证测试

- 验证可从界面创建一条合法任务。
- 验证任务详情页能完整展示全链路信息。
- 验证失败任务与成功任务的页面状态和提示明显不同。

### 步骤 24：实现告警、审计、配置与知识库页面

#### 执行指令

- 实现告警中心页、审计日志页、机械臂配置页、安全规则配置页、知识库管理页、示教样本管理页。
- 管理员页面必须具备权限保护。
- 审计日志页必须支持按任务、时间、机器人、状态检索。
- 配置页修改必须具备变更可追溯能力。

#### 交付物

- 告警中心页
- 审计日志页
- 配置管理页
- 知识库与示教样本页

#### 验证测试

- 验证审计日志检索结果准确。
- 验证配置修改后留下变更记录。

### 步骤 25：建立单元测试与集成测试基线

#### 执行指令

- 为后端核心模块建立单元测试。
- 为数据库迁移、任务流转、Planner、Coder、Critic、Worker、审计写入、告警生成建立集成测试。
- 覆盖正常流程与关键失败流程。
- 明确测试数据准备策略与清理策略。

#### 交付物

- 单元测试集
- 集成测试集
- 测试覆盖说明

#### 验证测试

- 验证所有核心模块都至少有基础测试。
- 验证成功流程、危险目标、校验失败、执行失败、复检失败都有自动化测试覆盖。
- 验证测试可重复执行且结果稳定。

### 步骤 26：建立前端端到端测试

#### 执行指令

- 使用 Playwright 建立控制台端到端测试。
- 至少覆盖登录、任务创建、任务查看、告警查看、审计查询、管理员配置访问六类流程。
- 端到端测试中使用模拟执行链路，不接真实硬件。
- 为关键页面增加加载态、失败态和权限态验证。

#### 交付物

- 前端端到端测试集

#### 验证测试

- 验证主要页面流转均可自动执行。
- 验证权限错误、接口失败、空数据场景都有页面级验证。
- 验证测试可在新环境中重复运行。

---

### 步骤 27：接入真实视觉设备前的灰盒验证

#### 执行指令

- 在接真实相机前，先验证视觉接口与标定数据结构、超时机制、异常结果处理方式。
- 检查视觉模块对异常帧、空帧、低置信度结果的处理是否保守。
- 确保任何视觉异常都不会推动任务进入真实执行。

#### 交付物

- 视觉接入前检查清单
- 视觉异常处理规则

#### 验证测试

- 验证空结果、低置信度结果、超时结果都会终止执行链路。
- 验证危险目标检测优先级高于目标定位成功。
- 验证视觉异常会产生日志和可追溯记录。

---

### 步骤 28：接入真实机械臂前的安全闸门验证

#### 执行指令

- 在接真实机械臂前，单独审查白名单动作、禁区配置、急停接口、执行超时、失败中断策略。
- 确认所有动作都来自模板映射，而不是自由生成。
- 确认任何未通过 Critic 的任务都无法进入 Robot 适配层。
- 确认急停路径不依赖 LLM。

#### 交付物

- 真实机械臂接入前安全检查清单
- 安全闸门确认记录

#### 验证测试

- 验证急停接口在模拟执行中可被稳定触发。
- 验证 Critic 拒绝的任务完全无法下发。
- 验证没有任何路径可以绕开白名单动作限制。

---

### 步骤 29：真实设备联调

#### 执行指令

- 先接真实相机，再接真实机械臂，禁止两者同时首次接入。
- 首轮只验证读取、识别、坐标变换、停机接口和非运动查询，不执行真实抓取。
- 第二轮只验证单个最小安全动作。
- 通过后再验证完整定点抓取放置闭环。

#### 交付物

- 联调记录
- 真实设备风险清单
- 闭环联调结果

#### 验证测试

- 验证真实相机结果与系统字段结构一致。
- 验证真实机械臂适配层返回的状态与模拟适配层一致。
- 验证最小动作、急停、完整抓取放置三类联调结果均有记录。

### 步骤 30：部署、运维与试运行验收

#### 执行指令

- 按 `tech_stack.md` 完成宿主机原生部署方案。
- 配置服务自启动、日志轮转、数据库备份、运行健康检查和错误恢复策略。
- 准备试运行验收清单，按设计文档中的测试场景逐条验收。

#### 交付物

- 部署说明
- 运维说明
- 试运行验收报告
- 已知问题与后续迭代清单

#### 验证测试

- 验证设计文档中的正常流程、安全流程、校验拦截、执行失败、记忆闭环、审计追溯六大验收场景全部通过。

## 4. 完成判定标准

只有同时满足以下条件，才允许判定 MVP 实施阶段完成：

- 所有步骤的验证测试全部通过。
- 没有跳过任何安全闸门步骤。
- 控制台、执行链路、审计链路、告警链路和长期记忆链路全部贯通。
- 真实设备联调结果与设计文档中的安全策略一致。
- 实现中未引入 `tech_stack.md` 已明确不推荐的复杂组件。

## 5. 对 AI 开发者的执行要求

- 每一步开始前，先复述该步骤目标、输入和输出。
- 每一步结束后，必须给出验证结果和剩余风险。
- 若验证失败，只能在当前步骤内修复，不允许带病进入下一步。
- 若发现设计文档与实际实现存在冲突，优先停止执行并更新问题清单，不自行扩大范围。
- 若必须做取舍，优先选择更安全、更简单、更可审计的方案。
