# 智能机械臂控制 Agent 系统 MVP 实施计划

## 当前实施澄清

- 本计划只以 `memory-bank/design_document.md` 与 `memory-bank/tech_stack.md` 作为实施基线。
- 当前 MVP 范围固定为：单工位、单品牌单型号机械臂、单机部署、Web 控制台、厂商 Python SDK 直连、校验通过后自动执行、异常立即停机并告警。
- 当前技术基线固定为：Windows 宿主机原生部署、Python 3.11、`uv`、FastAPI、Pydantic v2、SQLAlchemy 2.0、Psycopg 3、PostgreSQL 16、pgvector、React 19、Vite 8、TypeScript、Ant Design、pytest、Playwright。
- 大语言模型接入方式固定为共享模型 API 调用，必须保留 provider 抽象层、密钥隔离和模型调用审计。
- Agent runtime 固定为 `BaseAgent + Planner Agent + Coder Agent + Critic Agent + Executor Worker / Orchestrator`，不引入重型 Agent 编排框架。
- Tool 选择由 Agent 决策，Tool 路由、权限校验和安全门禁由 `Executor Worker / Orchestrator` 负责。
- 当前实现先使用模拟 `VisionTool` 与模拟 `RobotTool`，真实 YOLO 系列 ONNX 模型接入和真实机械臂接入保留为后续阶段，不作为当前实现完成标准。
- 审计链路必须覆盖“指令 -> 上下文装配 -> Tool 调用 -> 语义动作计划 -> 可执行结构化计划 -> 校验 -> 执行 -> 视觉复检 -> 记忆写入”，且不得持久化 raw chain-of-thought。
- 内部回退规则固定为：参数绑定错误、模板映射错误、脚本渲染错误优先回退到 `Coder Agent`；任务理解错误、动作顺序错误、安全前提不成立优先回退到 `Planner Agent`；完整内部循环最多允许 3 次。

## 1. 使用规则

- 只实现 `design_document.md` 已明确纳入 MVP 的能力，不自行扩展范围。
- 不引入 `tech_stack.md` 已明确不推荐的技术，包括但不限于：Milvus、Redis + Celery、Kafka、RabbitMQ、ROS、MoveIt、Kubernetes、LangChain/LlamaIndex 重编排、前后端多仓库。
- 所有执行链路实现必须优先满足安全、可审计、可回放、可定位问题，再考虑性能与扩展性。
- 任一步骤只有在本步骤验证通过后，才允许进入下一步。
- 在模拟链路未打通、规则校验未完成、急停路径未验证前，不允许接入真实机械臂。
- 在视觉异常处理、危险目标检测、复检判定未验证前，不允许接入真实视觉设备。
- 文档、代码、测试三者必须同步推进；如实现与设计冲突，先更新问题清单，再决定是否调整实现。

## 2. 总体实施顺序

实施顺序分为六个阶段：

1. 基础工程阶段
2. 后端基础能力阶段
3. Agent Runtime 与 Tool 阶段
4. Web 控制台阶段
5. 联调与测试阶段
6. 部署与验收阶段

只有当前阶段全部通过验证，才允许进入下一阶段。

- 急停规则说明

#### 验证测试

- 验证适配层只接受白名单动作和放行后的结构化计划。
- 验证非白名单动作和未放行计划会被拒绝。
- 验证模拟执行可返回全部标准状态并正确记录异常。

---

### 步骤 17：实现 Planner Agent

#### 执行指令

- 实现 `Planner Agent`，将自然语言任务转化为“语义动作计划”。
- Planner 只允许面向定点抓取放置场景输出受控语义步骤：定位目标物、接近目标物、闭合夹爪、提升目标物、转移到目标位置、放置目标物、松开夹爪、退出安全位置。
- Planner 允许请求 `VisionTool`，并在规划前判断危险目标、目标可定位性、目标是否落在允许工作区域内。
- Planner 不允许直接输出可执行脚本。

#### 交付物

- Planner 输入输出规范
- 语义动作计划格式

#### 验证测试

- 验证标准任务能够生成有限、清晰、可审计的语义动作计划。
- 验证检测到人手、危险物、目标不可定位、目标越界时，Planner 会直接失败并触发后续告警链路。
- 验证 Planner 不会输出设计范围之外的步骤。

---

### 步骤 18：实现 Coder Agent

#### 执行指令

- 实现 `Coder Agent`，将语义动作计划映射为“可执行结构化计划”。
- Coder 主要消费 `Memory Layer`，使用 SDK 白名单、动作模板、SOP 和示教样本进行参数绑定和脚本渲染，不直接请求 `RobotTool`。
- 输出必须保留：任务标识、步骤映射关系、动作参数、前置条件、脚本版本、生成时间、来源信息。
- 渲染出的 Python SDK 脚本仅作为执行工件和展示工件，不作为主数据源。

#### 交付物

- Coder 映射规则
- 可执行结构化计划格式
- 脚本工件格式说明

#### 验证测试

- 验证相同步骤输入得到一致的模板化输出。
- 验证超出白名单或模板边界的请求无法通过 Coder。
- 验证生成结果可追溯到任务、步骤、模板版本和来源上下文。

---

### 步骤 19：实现 Critic Agent

#### 执行指令

- 实现 `Critic Agent`，对语义动作计划、可执行结构化计划和渲染脚本进行执行前校验。
- Critic 可请求补充 `VisionTool` 以获取额外事实依据。
- 校验内容至少包括：脚本语法合法性、SDK 白名单、关节超限、位姿可达、工作禁区、基础防碰撞检查、夹爪动作阶段约束。
- 输出必须结构化包含：是否允许运行、失败类别、失败描述、关联步骤、建议回退模块。
- 回退规则严格按照设计文档执行，并计入最多 3 次内部循环上限。

#### 交付物

- Critic 校验规则
- Critic 结构化输出格式
- 回退矩阵说明

#### 验证测试

- 验证合法计划与模板可通过校验。
- 验证关节超限、不可达、禁区入侵、碰撞风险、非法夹爪阶段都会被拦截。
- 验证失败结果可明确定位到具体步骤，并给出正确的回退模块。

## 4. 完成判定标准

只有同时满足以下条件，才允许判定当前 MVP 实施阶段完成：

- 步骤 01 至步骤 28 的验证测试全部通过。
- 无硬件闭环、审计链路、告警链路和长期记忆链路全部贯通。
- 自动化测试覆盖正常流程与关键失败流程。
- 当前实现未引入 `tech_stack.md` 明确不推荐的复杂组件。
- 当前实现满足设计文档的单工位验收口径：`50` 个标准任务、`10` 类目标物、任务成功率 `>= 85%`、安全事故 `0`、视觉复检覆盖率 `100%`、失败任务拦截准确率 `>= 95%`、长期记忆写入正确率 `100%`、端到端耗时 `P95 <= 60 s`。
- 若当前阶段不接真实设备，则步骤 29 只保留为接入前门禁，不作为阻塞当前完成的硬性前置；若进入真实设备联调，则必须额外满足真实设备安全闸门要求。

## 5. 对 AI 开发者的执行要求

- 每一步开始前，先复述该步骤的目标、输入、输出和退出条件。
- 每一步结束后，必须给出验证结果、遗留问题和剩余风险。
- 若验证失败，只能在当前步骤内修复，不允许带病进入下一步。
- 若发现设计文档、技术栈文档与实际实现冲突，优先暂停并更新问题清单，不自行扩大范围。
- 若必须做技术取舍，优先选择更安全、更简单、更可审计、更易回放的方案。
